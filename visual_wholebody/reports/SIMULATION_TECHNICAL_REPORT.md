# 仿真系统技术报告

## 1. 范围与目标

本报告总结项目中 Isaac Gym 仿真系统的实现结构、URDF 到运行时数据结构的映射关系、关键 API 的输入输出语义，以及碰撞、接触、相机与对象资产的技术边界。报告覆盖 `high-level/` 与 `low-level/` 两条训练路径中的共性机制与差异化实现，重点回答“模型定义如何落到仿真执行”这一工程问题。

## 2. 仿真栈与代码入口

系统底层使用 Isaac Gym（Preview 系列）进行并行物理仿真。高层环境主入口为 `high-level/envs/b1z1_base.py` 及其任务子类 `high-level/envs/b1z1_pickmulti.py`，底层环境主入口为 `low-level/legged_gym/envs/manip_loco/manip_loco.py`。二者共享的核心流程是：加载 URDF 资产、创建 actor、绑定状态张量、循环执行控制与物理步进。

在运行时，仿真对象由 `gymapi.acquire_gym()` 获取 Gym 实例后建立，随后创建 sim、env、actor，并通过 `acquire_*_tensor` 与 `set_*_tensor` 在 PyTorch 张量与物理引擎之间交换状态与控制命令。这一流程决定了训练与推理的主要数据通道。

## 3. URDF 到仿真对象的映射机制

当前 B1Z1 URDF（高层与低层版本）定义了四足底盘、Z1 机械臂、夹爪与末端参考链接。URDF 中的 link 和 joint 在 `load_asset` 后被转换为 Isaac Gym 的刚体与关节索引体系。代码通过 `get_asset_dof_names/get_asset_dof_dict` 与 `get_asset_rigid_body_names/get_asset_rigid_body_dict` 建立“名称到索引”的语义映射，用于后续观测切片与控制下发。

需要区分两类状态：`default_joint_angles` 只描述可动关节的初始角（关节空间），不包含整机在世界坐标中的位姿；整机位姿与速度由 actor root state 管理，并在创建 actor 与 reset 时单独设置。这个分层使系统可以同时控制“机体在世界中的位置姿态”与“机体内部关节构型”。

URDF 中的末端参考点 `ee_gripper_link` 不是主要结构件，而是任务坐标锚点（tool frame）。它用于稳定地定义末端位姿、雅可比索引和 IK 目标，避免将控制语义直接绑定到可视网格几何中心。

## 4. 关节与自由度的工程语义

通用机器人学中，“可动关节数量”与“DOF 数量”不必相同（一个关节可有多个自由度）。但在当前 B1Z1 模型中，可动关节均为 1-DOF 的 revolute 关节，因此两者数值一致。腿部每条链路采用三个串联自由度（hip/thigh/calf），分别对应侧摆与前后摆/屈伸运动，属于将复合髋部运动工程化拆分为串联单自由度关节的常见做法。

在这种串联结构中，诸如 `FR_hip` 之类中间短连杆承担几何偏置与惯性表达作用，不必然承载碰撞体。机械臂中的 `link01/link05` 也属于类似的过渡段，主要服务运动学与动力学表达，而非关键接触建模。

## 5. 状态读取与控制输出接口

系统核心状态读取接口包括：

- `acquire_actor_root_state_tensor`：读取 actor 根状态（位置、姿态、线速度、角速度）。
- `acquire_dof_state_tensor`：读取关节位置与速度。
- `acquire_rigid_body_state_tensor`：读取刚体状态。
- `acquire_net_contact_force_tensor`：读取每个刚体的净接触力。
- `acquire_jacobian_tensor`：读取指定 actor 的雅可比。
- `acquire_force_sensor_tensor`：读取资产级力传感器（当前主要用于足端）。

控制侧关键接口包括：

- `set_dof_position_target_tensor`：下发关节位置目标。
- `set_dof_actuation_force_tensor`：下发关节驱动力/力矩。
- `set_actor_root_state_tensor_indexed`：按索引更新 actor 根状态。
- `set_actor_dof_states`：重置关节状态。

其中“名称语义”只在建映射阶段使用；真正执行控制时使用的是按索引对齐的连续张量。典型链路为：刚体名定位末端索引 -> 提取末端状态与雅可比 -> IK 求解关节目标 -> 组装全关节目标张量 -> 下发至 `set_dof_position_target_tensor`。

## 6. 碰撞建模与接触信号使用

URDF 中并非所有链接都定义碰撞体。当前模型采用“关键体积件建碰撞、部分过渡壳体省略碰撞”的策略，以降低计算开销和自碰撞误触发风险。该策略意味着：有碰撞体的部位发生接触时可被物理求解器约束；无碰撞体链接不会参与接触检测。

物理上，接触约束通常可抑制明显穿透，但并不意味着数值上零穿透。接触结果受时间步长、求解器参数和接触阈值影响。

训练中对接触信号的使用存在层级差异。低层 `manip_loco` 路径显式使用接触力构造碰撞惩罚（如 `_reward_collision`）并进入优化目标；高层路径虽读取接触张量和足端传感器，但主奖励中对机体/机械臂碰撞的显式惩罚较弱或未启用。换言之，低层更直接把碰撞作为训练约束，高层主要将其作为状态信号与辅助信息。

## 7. 相机系统与视觉输入

当前训练使用的相机不是在 URDF 中静态声明后自动接入，而是在 Python 环境初始化时动态创建并挂载到指定刚体。系统通过 `create_camera_sensor` 创建前视与腕视相机，并用 `attach_camera_to_body` 挂载到机器人链接。图像采集通过 `get_camera_image_gpu_tensor` 读取深度与分割张量，随后在环境内完成预处理、历史堆叠与观测拼接。

因此，URDF 负责机器人机械结构定义，视觉传感器则由环境代码与配置文件联合定义。这种解耦便于训练时快速改动相机位置、分辨率、视场角与随机扰动策略。

## 8. 多物体资产、名称语义与分割语义

高层多物体抓取任务由配置中的 `asset_multi` 驱动，包含 33 个对象类别（例如 `green_bowl`）。这些名称在环境中以对象列表形式保留，可用于任务统计和特征索引；但相机分割图通常提供的是类别/实例 id 通道，不天然携带自然语言标签。也就是说，“对象名称语义”来自环境元数据与配置映射，而不是来自相机像素本身。

对象资产目录中同时存在 `model.urdf`、`textured.obj`、`visual.ply` 与 `features.npy`。主训练流程明确使用了 `model.urdf`（仿真加载）与 `features.npy`（对象特征）；`visual.ply` 并非当前训练主路径的输入。

## 9. 网格真值点云能力边界

Isaac Gym 常规工作流提供的是相机观测（RGB/Depth/Seg）与动力学状态张量，不提供“直接返回任意对象网格真值点云”的一条龙 API。若需要无遮挡的网格真值点云，通常应从对象 mesh 文件离线采样生成。

仓库中的 `high-level/test_pointcloud.py` 即属于离线采样脚本：解析 OBJ 顶点与三角面，按三角形面积比例采样点并用 Open3D 可视化。该脚本用于点云检查，不参与主训练/评估调用链。

## 10. 结论

本项目仿真系统采用“URDF 结构定义 + 运行时张量接口 + 环境侧传感器/控制逻辑”的分层架构。URDF 提供机械拓扑、惯性与碰撞基础；Python 环境负责语义索引映射、相机与对象配置、观测组织、IK 与控制下发；Isaac Gym 提供并行物理步进与状态交换。

从工程效果看，该架构兼顾了训练吞吐、控制可解释性与任务扩展性。其关键边界在于：碰撞精度依赖碰撞体建模粒度，名称语义依赖环境配置映射，网格真值点云需离线资产采样而非仿真 API 直接给出。上述边界决定了系统在“可控性、可扩展性、真实性”之间的实际取舍。
